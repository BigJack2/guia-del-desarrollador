¿Qué es Git?

Git es un sistema de control de versiones distribuido de código abierto desarrollado por Linus Torvalds, el creador de Linux.

El control de versiones distribuido permite a los desarrolladores descargar un software, realizar cambios y subir la versión que han modificado. Todas las modificaciones subidas se guardan en versiones independientes, no sobrescribiendo en el archivo original.

La diferencia entre el control de versiones y Git, es que en Git cada desarrollador tendrá en el ordenador una copia del código fuente original y de las versiones disponibles del proyecto, permitiendo la ramificación y fusión.

De esta forma todos los desarrolladores interesados en el proyecto podrán ver las modificaciones realizadas y contribuir mejorando el código del Software.

Cuando nos referimos a la parte del hub hablamos de lo que hace especial a este Git, la comunidad e interacción con otros usuarios.

¿Qué es un control de versiones?

Un control de versiones permite a los desarrolladores administrar cambios en un software a la vez que el proyecto evoluciona.

En el caso de que un desarrollador quisiera trabajar en un proyecto, sería arriesgado realizar cambios sobre el código original. El control de versiones permite duplicar una parte de un proyecto de forma aislada y trabajar sobre ella sin que se modifique el repositorio original.

Una vez comprobado que el cambio se ha realizado con éxito, el desarrollador podrá fusionar su ramificación con el proyecto creando una nueva versión del mismo.

Esta nueva versión registra los cambios realizados sobre la versión anterior para que se puedan testear por otros desarrolladores.

Repositorio

Un repositorio es la ubicación o ruta en la que se almacena toda la información de un proyecto como imágenes, código, carpetas, documentos, etc.

Cada proyecto contaría con su propio repositorio único, por lo que la ruta de acceso será exclusiva para el proyecto.

Branch (ramificaciones)

En el caso de que queramos trabajar una parte concreta de nuestro proyecto de forma aislada no afectando al repositorio principal, tendremos que hacerlo mediante Branch.

El Branch creará una copia exacta de nuestro proyecto para hacer pruebas sin miedo a equivocarnos y que afecte a todo el trabajo realizado.

Pull Request (Fusión)

Cada vez que subas un nuevo cambio en una rama del proyecto, puedes avisar a los demás colaboradores para que validen o no tu pull request, o si encuentran posibles mejoras poder comentarlas.

Tag

Los Tag permiten controlar el estado de un repositorio dando información a otros usuarios de en qué versión se encuentra actualmente el proyecto.

Esta acción es conocida como “Tagging” y es bastante importante a la hora de gestionar la vida de un proyecto.

Crear un nuevo proyecto a partir de otro (Fork).

Fork

Una opción bastante usada en Github es la de Fork. Con esta opción podrás crear un nuevo proyecto en base a uno ya creado, permitiendo hacer modificaciones y guardándose en tu propio repositorio y no en el repositorio original.

Esta opción facilita el crecimiento de proyectos permitiendo a los desarrolladores continuar mejorando un software por cuenta propia y en el caso de realizarse una mejora en el repositorio principal podrás también implementarla a tu proyecto clonado.

Esta opción es conocida en Github como bifurcación.

Cuándo usar Github

Github es perfecto para gestionar desarrollos, permitiendo compartir los proyectos a una gran comunidad de desarrolladores que estarán encantados de cooperar y mejorar el software, aportando mejoras o comentarios que añadirán valor al desarrollo final.

Contar con un gestor de versiones como Git permite ver todos los cambios en la vida del proyecto desde sus inicios.

Git-Flow

En teoría tan agradable y luego la dura realidad golpea. Git-Flow intenta ser una estrategia de bifurcación universal que cubre muchos escenarios. La idea principal es tener un conjunto de sucursales que tienen una responsabilidad específica:

main/master: La versión actual en producción. Si necesita la versión de producción actual, simplemente consulte esta rama.

develop: La rama de desarrollo activo.

feature: una rama de característica se corta de develop, una vez que haya terminado, volverá a develop.

release: la rama de release es un guardián de la producción. Una vez que decidas que estamos listos para el release, te desconectas del develop y puedes hacer algunas correcciones de errores en esa rama. No se deben introducir nuevas características aquí.

hotfix - Hotfix: las ramas de Hotfix se cortan de main / master o release dependiendo de si desea hacer una corrección normal o una revisión en producción.

Si solo ve la imagen, es posible que se sienta abrumado. Para ser honesto, comprimí muchos casos de uso en una imagen, pero esas situaciones pueden surgir. El caso de uso típico si trabaja en equipos grandes o tiene que alinear los ciclos de soporte con otros equipos en su entorno. Un ejemplo típico sería un banco. Por lo general, no lanzan software semanalmente, por lo que todo debe estar alineado. Además, si trabajas en equipos grandes, Git-Flow escala bastante bien.

Ventajas

Fácil de alinear el trabajo incluso en varios equipos.

Manejo de múltiples versiones de productos.

Cada rama tiene una responsabilidad clara.

Tenemos una puerta entre la rama de desarrollo activo y la producción.

Contras

Un modelo complicado con muchas ramas. Eso también significa que tiene que fusionarse entre algunos de ellos, lo que aumenta la complejidad y también aumenta la posibilidad de conflictos de fusión.

Puede ralentizar el proceso de desarrollo/frecuencia de lanzamiento debido a todos esos pasos.

Se necesita una mayoría del equipo y la voluntad de apegarse a ella.

GitHub-Flow

Es una simplificación de Git-Flow en el sentido de que no tenemos ramas de lanzamiento. Tenemos una rama de desarrollo activa (a menudo main o master), que luego se envía a producción. Las funciones o las correcciones de errores se realizan a través de ramas de funciones de vida relativamente larga. La idea es que la rama main siempre pueda pasar a producción de una forma u otra. Normalmente, las personas tienden a trabajar con etiquetas para indicar si un compromiso específico se envía o no a un cliente.

La idea detrás de GitHub-Flow es fomentar un flujo de trabajo asíncrono. Eso significa que creamos algo y muchas veces tenemos una revisión que da como resultado comentarios, que luego deberían integrarse. La forma típica en que funcionan los proyectos de código abierto en GitHub (de ahí el nombre).

GitHub-Flow es perfecto para proyectos de código abierto incluso con un equipo de tamaño mediano. Fomenta los bucles de retroalimentación directa y el trabajo asíncrono. Para tales proyectos, también las ramas de características de larga duración son "normales", ya que las personas tienen menos tiempo para trabajar en esas cosas. Pero no me detendría ahí: este flujo de trabajo se adapta bien a muchos escenarios con un equipo pequeño donde todo está bajo su control. Si tiene que mantener varias versiones activas, lo más probable es que esta no sea la estrategia de ramificación que desea.

Ventajas

Ciclos de retroalimentación más rápidos y ciclos de producción cortos.

Perfecto para el trabajo asíncrono en equipos más pequeños.

Más fácil de entender que Git-Flow ya que es "más ágil".

Contras

Una vez que se fusiona una rama de características, está lista para la producción. Si no se prueba correctamente y está respaldado por un CI/CD fuerte, esto puede introducir errores más fácilmente.

Las ramas tienden a vivir más.

Puede volverse complicado para equipos más grandes (por ejemplo, mayor potencial de conflictos de fusión).

Admitir varias versiones de lanzamiento a la vez es muy difícil.

Gitlab-Flow

Gitlab-Flow se posiciona en algún lugar en el medio de Git-Flow y GitHub-Flow. En esencia, sigue siendo como GitHub-Flow pero agrega etapas antes de que pueda pasar a producción. Entonces, en lugar de decir main == production , Gitlab-Flow presenta aquí algunas ramas que reflejan sus entornos de ensayo "reales".

Ventajas

Puede manejar múltiples versiones de lanzamiento o incluso múltiples etapas.

Más simple que Git-Flow.

Se enfoca más en la calidad con un enfoque lean.

Contras

Puede volverse muy complejo si desea mantener varias versiones.

Más complejo que GitHub-Flow.

Trunk Based Development

El desarrollo basado en troncos ó "Trunk Based Development". Solo tiene una única rama compartida llamada troncal "Trunk". La idea es eliminar por completo las ramas longevas (para que podamos evitar fusiones infernales 😄). El diseño puede variar un poco. Hay dos sabores dependiendo del tamaño del equipo. Si el equipo es un poco pequeño, todos están directamente comprometidos con el tronco "Trunk". Una vez que estamos listos para enviar, creamos una rama release que refleja el estado en producción. Si el equipo es más grande, normalmente tiendes a crear también ramas de características de corta duración más pequeñas. En el mejor de los casos, integra incluso esas ramas de funciones con la mayor frecuencia posible. La idea aquí es que no implemente necesariamente toda la función, sino partes de ella para que pueda fusionarlas con frecuencia.

Contras

Si tiene un equipo sólido que comprende los requisitos y las implicaciones del desarrollo basado en toncos ó "Trunk Based Development", hágalo . No es realmente adecuado para proyectos de código abierto donde las personas tienden a tener ramas de funciones de larga duración debido a la naturaleza de la cantidad de proyectos de código abierto que funcionan.

Ventajas

Fomentar perfectamente las mejores prácticas de DevOps / Unit testing.

Mejora la colaboración.

Como casi no tenemos sucursales, hay menos posibilidades de conflictos de fusión.

Permite liberaciones rápidas.

Contras

Necesitas una especie de equipo senior para lograr esto. Debe saber cómo dividir una función para poder integrarla con regularidad. También fomenta la autonomía, lo que puede ser abrumador para los jóvenes.

Un fuerte sentido de CI/CD con pruebas útiles es imprescindible, de lo contrario, su rama troncal puede volverse inestable.
